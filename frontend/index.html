1. Read-Heavy (90% gets, 10% puts)

1
RocksDB

Combines a large block cache with one Bloom (or Ribbon) filter per SST file. In a point lookup the filter chain usually rules out every level but one, so the engine either (a) serves the value straight from cache or (b) touches exactly one 4 KB block on disk. The result is high throughput even when the data set exceeds RAM.

2
Redis

A GET is literally one hash and one pointer dereference, which lets Redis deliver about 36 k ops/s on a lone core and climb to just under 100 k ops/s when network I/O is overlapped by additional client threads. Throughput still trails RocksDB because every command must pass through a single event-loop thread, adding a few microseconds of dispatch overhead that RocksDB avoids.

3
Memcached

Although the data live entirely in DRAM, each lookup has to grab a hash-bucket or slab mutex, and that lock accounting shows up in the numbers: ~25 k ops/s with one worker and only ~56 k ops/s even under heavier load. The mutex traffic, plus an extra memory copy from slab to write buffer, trims enough microseconds from every request to keep Memcached well below both Redis and RocksDB.

4
MongoDB

Every key access still walks three to four cached B-tree nodes, so a single thread achieves roughly 14 k ops/s and all threads together reach the mid-40 k range. Those extra pointer hops, object metadata checks, and document locks consume more CPU per lookup than a hash or LSM read

2. Write-Heavy (10% gets, 90% puts)

1
RocksDB

An insert is just a WAL append plus a memtable write—both sequential, lock-free operations. Large in-memory write batches and background compaction keep the foreground path short, so throughput stays well ahead of every other engine at every thread count.

2
Redis

Writes are in-RAM pointer assignments; without an immediate fsync, the critical path is only hash computation and object creation. Although a single event loop serialises commands, that loop is efficient enough that Redis outpaces Memcached in three of the four thread settings.

3
Memcached

Pure memory inserts keep base latency low, but each mutation briefly grabs both a hash-bucket lock and a slab-class lock, adding overhead absent in Redis. Those mutex costs hold Memcached just below Redis except at the four-thread mark, where extra workers momentarily tip it ahead.

4
MongoDB

Every put traverses multiple B⁺-tree levels, may trigger leaf-page splits, and must be durably journaled. That heavier per-write work leaves MongoDB well behind the in-memory engines and far below RocksDB's append-only path in all configurations.

1
Redis

When reads and writes are both O(1) RAM operations, adding the other half of the workload hardly hurts—until memory pressure forces evictions. With the data volume used here Redis stays comfortably in memory and tops the chart.

2
RocksDB

Each read is still cache-plus-filter fast and each write is still append-only, but the MemTable flush frequency increases relative to the read-heavy run. Those extra background flushes explain why it trails Redis in the mixed load.

3
Memcached

Lock contention now comes from both gets and puts, so the effective hit on concurrency is bigger than in either extreme workload, pushing the bar below RocksDB.

4
MongoDB

The engine must interleave random writes, journal syncs, and buffer-cache reads. Cache churn (constant eviction and re-loading of data blocks in a cache because the working set is larger or “busier” than the cache can comfortably hold) and page pins (when the engine temporarily locks a memory page—typically while a writer is modifying it—so the page cannot be evicted or reused until the pin is released) cut into both sides of the 50/50 mix, leaving it last.

1
RocksDB

SST files are immutable, fully sorted runs. Once a scan starts, the iterator streams large contiguous blocks, which modern SSDs pre-fetch efficiently. Levelled compaction keeps key ranges from spanning many files, so very few overlaps have to be merged on the fly.

2
MongoDB

B⁺-tree leaves are physically linked. After locating the first key, the cursor walks those links sequentially. It is fundamentally a good fit for scans, but each hop still means a pointer chase and an occasional uncached page, hence ~2.5× slower than RocksDB's flat-file streaming.

3
Redis

The native key space is hash-distributed, so a true lexicographic range requires a full iteration (SCAN) over all keys. Only a small test data set keeps this from dropping to single-digit kops/sec.

4
Memcached

Provides no ordered data structure whatsoever; a range query degenerates into "dump every key then filter client-side," which is prohibitive. The benchmark therefore records effectively zero useful throughput.

Threading Scalability Analysis

Understanding Scaling Efficiency

What is Scaling Efficiency?

Formula: (Actual Throughput) ÷ (Single-Thread Performance × Thread Count) × 100%

Measures how well each database utilizes additional threads compared to perfect linear scaling.

100% Efficiency = Perfect Linear Scaling

Each additional thread provides full benefit. Example: 1 thread = 25K ops/sec → 4 threads = 100K ops/sec

< 100% Efficiency = Scaling Bottlenecks

Diminishing returns due to lock contention, memory bandwidth limits, or cache conflicts

> 100% Efficiency = Super-linear Scaling

Rare but possible due to better cache utilization or reduced per-thread overhead

Why This Matters

  • • System Design: Identify when adding threads helps vs. hurts
  • • Resource Planning: Find optimal thread counts
  • • Bottleneck Detection: Reveal architectural limitations
  • • Workload Impact: See how different operations scale differently

Key Insights

• Perfect scaling is rare: Most systems show diminishing returns

• Workload matters: Same database scales differently for reads vs. writes

• Architecture impacts: Hash tables, B+ trees, and LSM trees have different scaling characteristics

Thread Scaling Efficiency by Workload

Balanced workload (50% reads / 50% writes)

Rank: MongoDB > Memcached ≈ Redis > RocksDB
1
MongoDB

Document-level locks let several writer and reader threads proceed in parallel, and its group-commit journal amortises fsync, so it preserves the largest fraction of ideal linear speed-up.

2
Memcached

Benefits from extra worker threads until hash-bucket and slab-allocator mutexes turn hot, giving it middling—but still positive—returns.

3
Redis

Keeps all command execution in a single event loop; additional client threads only queue, so efficiency falls.

4
RocksDB

Drops to last because the 50-50 mix doubles memtable flushes and triggers frequent background compactions that steal cache lines and I/O bandwidth from foreground threads.

Read-heavy workload (90% reads / 10% writes)

Rank: MongoDB > Memcached ≈ Redis > RocksDB
1
MongoDB

With most requests satisfied out of the WiredTiger cache, MongoDB's pointer-chase pipeline scales well until shared cache traffic saturates.

2
Memcached

Gains from multiple readers but still pays for brief hash-bucket locks, so its curve sits a little lower.

3
Redis

Again suffers from its single execution core, making efficiency decay quickly.

4
RocksDB

Finishes last because every extra reader hammering the block cache and Bloom-filter metadata causes contention on shared CPU caches and memory bandwidth, eroding its theoretical concurrency advantage.

Write-heavy workload (90% writes / 10% reads)

Rank: MongoDB > Memcached ≈ Redis > RocksDB
1
MongoDB

Document-level locks mean writers rarely block each other, and group-commit batches many updates into one journal flush; extra threads therefore convert into real work, giving the best scaling curve.

2
Memcached

Inserts live in RAM, yet every write still grabs a hash-bucket and slab lock; contention on those brief critical sections trims—but doesn't erase—the benefit of more worker threads.

3
Redis

All commands run in a single event loop; additional client threads only feed that loop faster, so once one core saturates, throughput flat-lines and efficiency falls.

4
RocksDB

With 90% writes, the memtable fills often, triggering frequent flushes and compactions; those background jobs steal I/O and stall writers, producing the steepest drop in per-thread efficiency.

Range-query workload (95% scans / 5% inserts)

Rank: MongoDB > RocksDB > Redis (Memcached not applicable)
1
MongoDB

Each scanning thread walks its own leaf-page chain, and document-level locks seldom collide, so adding cores translates into almost linear gains—highest core-scalability in the range workload.

2
RocksDB

Extra cores help until shared block-cache and merge buffers saturate; the need to reconcile results from overlapping SST levels limits per-core benefit, placing its scalability below MongoDB's.

3
Redis

The single command loop handles every SCAN; once one core is saturated extra cores sit idle, so scalability plunges—efficiency drops most sharply after the first thread.

Latency Analysis

Latency Percentiles by Workload

1. Balanced workload (50% reads / 50% writes)

Observed Order: RocksDB < Redis < Memcached < MongoDB
1
RocksDB

Most requests are answered from the block cache or the memtable, yielding sub-10 µs averages and tightly clustered tails. Compaction threads occasionally pre-empt cache lines, so the 99th-percentile ticks up to ~20 µs, but the gap remains modest.

2
Redis

Command execution is single-threaded but fully in RAM, so the mean is only ~70 µs. Occasional AOF fsyncs or hash-table rehashes introduce ~2× jumps at P95 and P99.

3
Memcached

Additional worker threads introduce short lock waits on hash buckets and slab metadata; these pushes the 95th and 99th percentiles to ~2.5× the mean.

4
MongoDB

Each operation triggers a B⁺-tree traversal and, for writes, a journal append. Cache misses or group-commit flushes extend some requests beyond 500 µs, creating the tallest red bar in this panel.

2. Read-heavy workload (90% reads / 10% writes)

Observed Order: RocksDB < Redis < Memcached < MongoDB
1
RocksDB

Point look-ups that pass the Bloom filter are completed from memory in ~8 µs on average; only cache misses elevate tail latencies to ~30 s.

2
Redis

Reads remain an O(1) hash probe, but periodic eviction and AOF activity stretch the 99th percentile to ~180 s.

3
Memcached

Mean latency sits near Redis, yet lock contention on heavily read buckets broadens the tail to ~290 s.

4
MongoDB

Every miss forces a random 4 KB SSD read; when several threads collide on the I/O queue, tail latency rises above 450 s.

3. Write-heavy workload (90% writes / 10% reads)

Observed Order: RocksDB < Redis < Memcached < MongoDB
1
RocksDB

Inserts land in the memtable and WAL, both sequential, so average latency remains in single-digit microseconds; background compaction is decoupled, keeping the tail below 25 µs.

2
Redis

The fork-based AOF writer flushes once per second (fsync-everysec), producing occasional stalls that widen P95 and P99 to roughly double the mean.

3
Memcached

Each mutation locks a slab class and a hash bucket; as update pressure rises, those locks inflate the 99th percentile to >300 µs.

4
MongoDB

Page splits and synchronous journal writes dominate; worst-case requests exceed 500 µs, explaining the pronounced red bar.

4. Range-query workload (95% scans / 5% inserts)

Observed Order: RocksDB < MongoDB < Redis
1
RocksDB

Each SSTable is already sorted, so a range iterator can stream large, contiguous blocks from disk. Leveled compaction keeps overlap low, meaning the scan seldom has to merge more than a couple of files at once. As a result, both the average and the 99ᵗʰ-percentile stay well under 1 ms.

2
MongoDB

Leaf pages in its B⁺-tree are linked sequentially; once the first key is found the cursor follows those links in order. However, every page hop still involves pointer chasing and may trigger an uncached 4 KB read, so tail latency rises into the low-millisecond range—slightly above RocksDB's flat-file stream but still respectable.

3
Redis

The native key space is hash-distributed, so there is no intrinsic order. A true range therefore forces a SCAN over all buckets while the single event loop remains busy. This full-space walk produces long pauses, pushing the 95ᵗʰ and 99ᵗʰ percentiles into multi-millisecond territory and giving Redis the poorest latency in this workload.

Use Case Analysis

Understanding how each database engine's architectural choices create specific performance characteristics and determine optimal use cases.

RocksDB — Log-Structured Merge Tree

Relative Strengths

Outperforms the other engines on point reads, writes, and range scans.

Combines Bloom-filtered look-ups, an in-memory memtable, and sequential SST streaming to minimise both seek count and latency.

Relative Weakness

Displays the steepest decline in scaling efficiency once several foreground threads and background compaction threads compete for shared cache lines and I/O bandwidth.

Best-fit Workload

Well suited to data-ingest pipelines that are dominated by high-volume, append-only writes—examples include metrics collectors, time-series databases, and log aggregation tiers. In such systems the write path is the performance gate, while read access is typically limited to occasional point look-ups from monitoring dashboards. RocksDB's superior single-thread ingest rate and low-latency reads directly satisfy these needs, and its imperfect multi-core scaling is largely irrelevant because ingestion is often handled by a handful of long-lived writer threads rather than hundreds of concurrent clients.

Redis — In-Memory Data-Structure Store

Relative Strengths

Provides microsecond-level latency for both reads and writes as long as the data remain resident in RAM.

Offers lightweight persistence controls, allowing the write path to remain short when durability can be relaxed.

Relative Weaknesses

Throughput plateaus quickly once additional CPU cores are introduced because all commands execute in a single event loop.

Lacks intrinsic key ordering, forcing range queries to devolve into full key-space scans.

Best-fit Workload

Ideal for front-end caching layers, user-session stores, real-time counters, or leaderboards where every request must complete in microseconds and the entire working set is expected to fit comfortably in memory. These workloads issue almost exclusively point operations, tolerate relaxed durability, and are typically sharded by client libraries, so neither ordered range queries nor multi-core scalability are critical. Redis's latency edge therefore translates directly into perceivable application responsiveness.

Memcached — In-Memory Hash Cache

Relative Strengths

Achieves point-operation latency only modestly slower than Redis, while remaining entirely stateless and free from persistence overhead.

Avoids any write-ahead logging, eliminating fsync stalls.

Relative Weaknesses

Hash-bucket and slab-allocator mutexes become hot with more than a handful of worker threads, limiting scaling benefits.

Offers no native mechanism for ordered scans or durable storage.

Best-fit Workload

Excels as a transient edge cache in front of a durable data store—for example, CDN metadata, web-session tokens, or API throttling keys. In these scenarios, occasional data loss is acceptable, operational simplicity is paramount, and the application tier already handles sharding. Because requests are dominated by single-key look-ups and the cache layer is not expected to scale linearly with core count, Memcached's modest latency advantage over disk-backed systems outweighs its scaling ceiling and functional limitations.

MongoDB — WiredTiger B⁺-Tree Engine

Relative Strengths

Retains the highest fraction of ideal speed-up as threads increase, owing to document-level locks and batched journal commits.

Delivers range-scan performance second only to RocksDB, aided by sequential leaf-node links and integrated secondary indexes.

Relative Weaknesses

Exhibits the slowest point-operation latency and the lowest single-thread throughput, as each request traverses multiple B-tree levels and incurs journal overhead.

Best-fit Workload

Best matched to document-centric OLTP or micro-service back-end workloads that value flexible JSON schemas, secondary-index range filters, and multi-document transactional guarantees. Such systems are typically deployed on multi-core servers and serve many concurrent users, so MongoDB's superior scaling efficiency and ordered-scan capability provide tangible benefits, while its higher per-operation latency is masked by horizontal scale and application-level caching.
